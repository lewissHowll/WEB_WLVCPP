# WLVCPP web app

A web front end for Bahaa Ismail's WLVCPP cell-penetrating peptide (CPP)
predictor — FastAPI backend + Vue 3 frontend, wrapping the original
`WLVCPP.py` algorithm.

## What changed from the original script

1. **Models are trained once, not per request.** The original CLI script
   retrained both SVMs from scratch on every run (~8s for a handful of
   peptides on this container, since it refits on 820 peptides twice).
   `backend/train_models.py` trains once and pickles the models
   (`backend/models/*.joblib`); the API just loads and scores. This should
   drop response time to well under a second.
2. **Fixed batch-dependent scoring — the biggest algorithmic change.**
   The original `categorise()` binned each query peptide's features into
   deciles based on quantiles of *whatever else was in the same request*,
   so the same peptide could score very differently depending on batch
   composition (demonstrated: the TAT peptide scored 84.2% alone vs 98.3%
   alongside a second peptide). Scores are now computed against fixed
   reference bucket boundaries learned once from the training data
   (`fit_reference_stats()` in `backend/wlvcpp/features.py`), so a
   peptide's score no longer depends on what else you submit alongside it.
   **This changes prediction outputs from the original script** — see
   `docs/reference_validation.md` for the full writeup and accuracy
   comparison against `rand_test.fa`/`recep_test.fa` before deploying this.
   Short version: the new approach is both more consistent *and* slightly
   more accurate than the original, including in the original's own
   best-case scenario. `backend/validate_reference_fix.py` reproduces the
   numbers.
3. Peptide entries are capped at 500 per request (`MAX_ENTRIES_PER_REQUEST`
   in `backend/main.py`) as a basic abuse guard — raise if you need more.
4. **Added password protection.** The whole site (frontend + API) sits
   behind HTTP Basic Auth — see the "Password protection" section below.

## Password protection

The entire site — frontend and API alike — is gated behind HTTP Basic Auth
(`backend/auth.py`), controlled by two environment variables:

- `APP_USERNAME`
- `APP_PASSWORD`

**If both are unset, auth is disabled** — this keeps local dev friction-free
(a warning is logged loudly at startup so it's obvious). For any real
deployment, set both.

`/api/health` is always left open regardless, so uptime monitors don't need
credentials.

**Locally:**
```bash
APP_USERNAME=labuser APP_PASSWORD=choose-a-real-password uvicorn main:app --port 8000
```
Your browser will show a native login prompt on first visit.

**On Render:** Dashboard → your service → Environment → Add Environment
Variable, add both `APP_USERNAME` and `APP_PASSWORD`. Render redeploys
automatically when you save. Pick a real password, not the example above —
it's sent base64-encoded (trivially reversible, not encrypted) so this only
protects you as long as the site is served over HTTPS, which Render does by
default.

This is one shared login for the whole research group, not per-user
accounts — fine for "keep the public out," not a substitute for real auth
if you ever need to distinguish between individual users.

## Project layout

```
backend/
  wlvcpp/            parse.py, features.py, constants.py, predict.py
  data/              rand_train.fa, recep_train.fa, rand_test.fa, recep_test.fa
  models/            *_model.joblib, *_refstats.joblib — generated by train_models.py, not hand-edited
  train_models.py    run this whenever the training FASTA files change
  validate_reference_fix.py   compares old vs new scoring against the held-out test sets
  main.py            FastAPI app (POST /api/predict, GET /api/health)
  requirements.txt
frontend/
  src/App.vue         main UI
  src/components/     SequenceStrip.vue — colour-codes residues by group
  src/aminoAcids.js    mirrors the backend's amino-acid grouping
docs/
  reference_validation.md   write-up of the batch-dependence fix, for the researchers
Dockerfile            builds frontend, trains models, serves both from one container
```

## Run locally

**Backend:**
```bash
cd backend
pip install -r requirements.txt
python train_models.py      # generates models/*.joblib — do this once
uvicorn main:app --reload --port 8000
```
(To run the test suite locally: `pip install -r requirements-dev.txt && pytest tests/ -v`.)

**Frontend (dev mode, hot reload):**
```bash
cd frontend
npm install
npm run dev                 # proxies /api to localhost:8000, see vite.config.js
```

**Or build the frontend into the backend and serve everything from one process:**
```bash
cd frontend && npm run build   # outputs into ../backend/static
cd ../backend && uvicorn main:app --port 8000
```
Then open http://localhost:8000 — the API and UI are on the same origin.

## Deploying (Render.com)

1. Push this repo to GitHub.
2. In Render: New → Web Service → connect the repo.
3. Environment: Docker (it'll pick up the `Dockerfile` automatically).
4. Set `APP_USERNAME` and `APP_PASSWORD` in the Environment tab (see
   "Password protection" above) — do this before the first deploy if you
   don't want the site briefly public.
5. Render builds the image (which runs `train_models.py` at build time) and
   starts the container — first request should be fast since models are
   already trained.

If `rand_train.fa` / `recep_train.fa` ever change, just push — the Docker
build retrains automatically since `train_models.py` runs during `docker build`.

## CI/CD

`.github/workflows/ci.yml` runs on every push and PR to `main`:

- **backend job** — installs deps, trains the models, runs `pytest` (smoke
  tests including a regression check that the well-known TAT peptide still
  scores as CPP), then boots the API and hits `/api/predict` for real.
- **frontend job** — `npm ci` + `npm run build`, so a broken build fails CI
  before it ever reaches Render.
- **deploy job** — only on pushes to `main`, only if both jobs above pass.

You have two options for the actual deploy, pick one:

1. **Simplest — Render's own auto-deploy.** Connect the repo in Render and
   leave "Auto-Deploy" on. Render redeploys on every push to `main` by
   itself; you don't need the `deploy` job or any secret at all. CI still
   runs and gives you a pass/fail signal, it just won't block the deploy.

2. **CI-gated deploy.** If you want Render to *only* deploy after CI passes
   (rather than deploying immediately on push, independent of test
   results), turn off Render's auto-deploy and instead:
   - In Render: Settings → Deploy Hook → copy the URL.
   - In GitHub: Settings → Secrets and variables → Actions → New repository
     secret → name it `RENDER_DEPLOY_HOOK_URL`, paste the URL.
   - The `deploy` job will then curl that hook after CI passes. If the
     secret isn't set, the job just logs a message and exits cleanly rather
     than failing — so it's safe to merge this workflow before you've
     decided which option to use.

```json
{ "fasta": ">my_peptide\nKWKLFKKIEKVGQNIRDGIIKAGPAVAVVGQATQIAK\n" }
```
